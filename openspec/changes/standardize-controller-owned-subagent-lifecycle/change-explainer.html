決策焦點

把 controller 變成 node lifecycle 與資源權威，讓 adapter 回到正式觀察路徑。

這個 change 將目前混在 controller/adapters 內的大量例外恢復分支，拆成「正式路徑」與「例外處理器」。 Controller 先建立或驗證 branch、worktree、session，再讓 subagent/runner 加入；adapter 只回報標準化 observation； abnormal observation 則交給 ControllerExceptionHandler 與可治理的 recovery rule/playbook。

Change: standardize-controller-owned-subagent-lifecycle Capability: controller-subagent-lifecycle Decision: Controller owns DAG node lifecycle and resources; adapters attach to prepared resources and report normalized observations; abnormal observations go to ControllerExceptionHandler; repeated signatures produce auditable recovery-rule/playbook proposals; completion remains controller-validated and integrated. Success signal: nodes expose controller-owned acceptance/resource/runner/observation/judgment/validation/integration phases; retries do not create duplicate uncontrolled branches/worktrees/sessions.

Why

目前 controller/subagent orchestration 把正式流程與大量 exception-specific recovery 混在一起。這些 recovery 曾解決真實事件，但也讓 controller loop 難以推理，並可能建立 duplicate sessions 或 replacement attempts，導致有用上下文流失。

此 change 的壓力來自使用者明確決策：controller 應成為 node lifecycle 與 resource inventory 的權威；adapter 走正式 observation path；abnormal recovery scripts 應移到獨立 exception handler。

What Changes

  • 新增 controller-owned DAG node lifecycle：acceptance、resource initialization、runner join/start、adapter observation、controller judgment、validation、integration、terminal closeout。
  • branch/worktree/session ownership 移入 controller lifecycle；subagent/adapter 在正式路徑只 attach 到 prepared resources。
  • adapter 回報 normalized observations；exception handler 決定 protocol violation、runner error/lost、stale/unresolved、blocked 等恢復動作。
  • 新增 controller recovery policy/playbook：deterministic rules 優先，unknown/repeated failures 可 invoke controller model 並產生 auditable recovery-rule proposal。

Affected Surfaces

  • Core：src/core/controller-loop.ts、src/core/subagent-adapter.ts、src/core/types.ts、src/core/git-workspace.ts。
  • Adapters：src/adapters/pi/*、src/adapters/opencode/*。
  • Data/contracts：schemas、durable DAG/subagent state、controller ledger events、recovery decision records、native-git workspace allocation semantics。
  • Operator surfaces：monitor/status 顯示 lifecycle phase、resource ids、abnormal observation、recovery decision、learned-rule id（欄位選擇仍是 open question）。

Scope Boundaries

In scope：controller-owned lifecycle、prepared-resource adapter attach、normalized observation、exception handler、recovery-rule/playbook proposal、validation/integration gate preservation。

Out of scope：unsafe one-shot removal of all legacy recovery behavior、subagent-created branches/worktrees、direct unreviewed source edits by LLM、model-visible /goal tool redesign、unrelated DAG planning semantics。

Rollout

  1. Add lifecycle/resource/recovery types, schemas, and store fields additively.
  2. Introduce prepared-resource adapter capability with legacy startSession compatibility.
  3. Route Pi/OpenCode behind feature flag or compatibility shim.
  4. Extract existing recovery branches into default exception handler parity decisions.
  5. Add controller-model diagnostic and learned-rule proposal support.
  6. Switch controller loop to lifecycle-first execution after parity tests pass.
  7. Deprecate legacy resource-creating adapter starts after adapters/tests migrate.

Before / After：評審正在批准什麼改變

Before：正式路徑與例外分支混在一起

  • Ready node 透過 workspace allocator 與 adapter.startSession() 啟動。
  • Pi/OpenCode adapter 啟動 detached runner 並從 transcript 推論狀態。
  • Controller loop 同時處理 validation、integration 與 transient/quota/context/missing-session/terminated/stale 等 hard-coded recovery branches。
  • Replacement session/workspace 可能造成 duplicate resources 或遺失有用上下文。

After：controller lifecycle + adapter observation + exception handler

  • Controller 先定義 acceptance，再建立/驗證 branch、worktree、session、model routing。
  • Subagent/adapter 僅 attach/start 到 controller-prepared resources。
  • Adapter 只回報 normalized observations，例如 selfReportedComplete、protocolViolation、runnerLost。
  • Exception handler 根據 deterministic rules、controller-model diagnostic、learned rule proposals 做恢復決策。

決策點、取捨與建議

D1. Controller owns lifecycle/resources

Recommendation: Accept

選擇：新增 durable node execution lifecycle，並保留 coarse status projection 供 migration。

取捨

狀態模型較大，但能避免資源權責分散。

拒絕方案

只擴充現有 enum；只保留 workspace hook。

D2. Adapter formal observation

Recommendation: Accept

選擇：adapter attach/start prepared resources，並回報 facts/protocol observations，不做 recovery policy。

取捨

需要新 contract，但可讓 Pi/OpenCode 共用 controller policy。

拒絕方案

把 exception logic 放在 adapter；沿用混合式 failed/needsFollowup。

D3. Exception handler owns recovery

Recommendation: Accept

選擇：異常 observation 進 ControllerExceptionHandler，輸出 durable RecoveryDecision。

取捨

多一層抽象，但主流程與例外恢復分離。

拒絕方案

繼續在 controller-loop 增加 pattern branches；所有情況都問模型。

D4. Repeated failures → rule proposals

Open activation policy

選擇：重複 signature 產生可審核 recovery-rule/playbook proposal，不直接改 source。

取捨

學習能力需 governance；避免未審核 runtime source edit。

待決

自動啟用還是 human review 後啟用仍是 open question。

D5. Recovery reuses resources

Recommendation: Accept

選擇：預設重用 controller-owned worktree/branch/session；新資源需 supersession reason。

取捨

保留 context，但可能保留 dirty/bad state。

緩解

exception handler 需記錄 dirty state 並可 ask user/supersede。

D6. Completion stays validated

Recommendation: Accept

選擇：SUBAGENT_RESULT 只進 controllerJudging，仍需 validation/integration。

取捨

完成路徑較長，但維持 self-report 不可信任的安全邊界。

拒絕方案

把 subagent self-report 視為 terminal complete。

Implementation slices 與 acceptance criteria

Slice 1 — Spec / Contract foundation

定義 lifecycle phase、prepared resource、adapter observation、exception handler、RecoveryDecision、recovery rule/playbook。

  • Acceptance：schemas/store/contracts 可表示 lifecycle/resource/recovery evidence 且不破壞既有 status consumers。

Slice 2 — Controller-owned lifecycle

加入 lifecycle transitions、coarse status projection，並把 node branch/worktree/session preparation 移入 controller lifecycle step。

  • Acceptance：ready node 在 subagent work 前會記錄 acceptanceDefined、resourcesCreating、resourcesReady、runnerStarting、runnerActive。

Slice 3 — Adapter formal path

Pi/OpenCode 支援 prepared-resource attach/start，adapter inspection emit normalized observations，保留 formal markers parsing。

  • Acceptance：adapter 不做 context/model fallback、replacement、terminal blocking 等 recovery policy decision。

Slice 4 — Exception handler / recovery policy

抽出既有 recovery 行為成 parity decisions，新增 deterministic rule lookup、controller-model diagnostic、rule proposal persistence。

  • Acceptance：protocolViolation、runnerError、runnerLost、stale/unresolved、self-reported blocked 皆透過 exception handler 並持久化 decision。

Slice 5 — Validation / integration / closeout

保留 selfReportedComplete → controllerJudging → validating → integrating → complete；closeout/cleanup 使用 controller-owned resource records。

  • Acceptance：required integration gates 仍阻止未整合 node complete。

Slice 6 — Monitor / documentation / tests

更新 monitor/status、README、adapter contract，補 lifecycle、adapter observation、exception-handler parity、no duplicate resources 等測試。

  • Acceptance：`npm run check` 通過，source manifest refresh/validation 通過。

Risk register 與 high-risk filter

Lifecycle state model is too large for one change High

Mitigation：additive implementation、compatibility projections；必要時拆 follow-up。

Controller-model diagnostics make recovery nondeterministic Medium

Mitigation：persist evidence/decision、prefer deterministic rules、bounded actions、validation gates。

Generated rules could encode a bad recovery High

Mitigation：activation state、confidence、source evidence、validation requirements。

Prepared-resource attach may not map cleanly to every harness Medium

Mitigation：adapter capability negotiation 與 legacy fallback。

Reusing workspaces can preserve bad dirty state Medium

Mitigation：exception handler inspect/record dirty state，可 ask user 或 supersede with evidence。
