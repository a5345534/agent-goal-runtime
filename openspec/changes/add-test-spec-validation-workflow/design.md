## Context

The runtime already has the core pieces needed for orchestration: DAG nodes, subagent registry records, controller polling, validation runner hooks, and durable stores. The recent fail-closed validator change prevents a declared validator from being silently skipped and still passing. However, this still assumes that a good validator exists before implementation completion.

The desired generic pattern is test-spec-first orchestration:

```text
objective
  -> test-spec writer creates tests/validators
  -> reviewer/controller approves the test contract
  -> implementation subagent works against approved tests
  -> controller reruns validators and checks locked tests
  -> final audit either accepts, sends implementation back, or identifies a test gap
```

The runtime should not become a planner or test generator. It should consume a DAG produced by `agent-goal-planner` (or hand-authored JSON), enforce the validation contract encoded in that DAG, persist evidence, and expose enough operations for test-gap revision loops.

## Goals

- Make test/validator contracts first-class DAG metadata without breaking existing DAGs.
- Ensure high-risk implementation work cannot complete from self-report alone.
- Ensure declared validators must execute or fail closed.
- Ensure implementation subagents cannot weaken approved test artifacts unnoticed.
- Persist validation evidence so monitor/status/final audit can show what actually ran.
- Give final audit a structured way to classify failures as either implementation bugs or test gaps.
- Keep planner and project rule-pack generation outside runtime.

## Non-Goals

- Runtime will not generate tests with an LLM.
- Runtime will not hard-code repository-specific package, module, or build rules.
- Runtime will not guarantee that tests are semantically complete; it will enforce the review/lock/rerun workflow and record evidence.
- Runtime will not require every low-risk docs node to use the full test-spec workflow.

## Proposed DAG Model

Existing fields remain valid:

```json
{
  "id": "implement-user-role-move",
  "validators": ["npm test"],
  "expectedOutputs": []
}
```

New optional metadata is added under a single validation contract object so older DAGs keep working:

```json
{
  "id": "implement-user-role-move",
  "kind": "implementation",
  "risk": "high",
  "validation": {
    "profile": "code-move",
    "testSpecNodeId": "test-spec-user-role-move",
    "approvedByNodeId": "review-test-spec-user-role-move",
    "artifactLocks": [
      {
        "path": "openspec/changes/user-role-move/validators/check.sh",
        "sha256": "...",
        "sourceNodeId": "test-spec-user-role-move"
      }
    ],
    "requiredEvidence": [
      "validators-ran",
      "locked-artifacts-unchanged",
      "implementation-diff-present"
    ],
    "onAuditTestGap": "reopen-test-spec"
  }
}
```

### `kind`

Suggested values:

- `test-spec` — writes or revises tests/validators/acceptance specs.
- `test-review` — reviews test-spec output; often a controller/review model node.
- `implementation` — makes product/code/docs changes that must satisfy an approved contract.
- `audit` — reviews completed work and may identify implementation bugs or test gaps.

Runtime treats unknown kinds as generic execution nodes unless validation policy requires otherwise.

### `validation.profile`

A symbolic profile generated by planner/rule packs. Runtime persists and displays it, but does not expand project-specific rules. Generic built-in profiles may contribute built-in checks such as:

- `code-change`: require implementation diff and validators.
- `code-move`: require implementation diff, old references removed, owner destination present (concrete commands come from rule pack).
- `docs-spec-change`: require spec/docs files changed and validators.
- `lint-hook`: require executable hook and self-test command.
- `audit-report`: require report artifact and audit evidence.

### `artifactLocks`

Approved test artifacts are hashed after the test-spec/review phase. Implementation validation checks the hashes before accepting completion. If the implementation subagent changes locked artifacts, validation fails with a test-contract mutation error unless the node explicitly references a newer approved test-spec revision.

### `requiredEvidence`

A node may require generic evidence labels. Initial labels:

- `validators-ran`
- `locked-artifacts-unchanged`
- `implementation-diff-present`
- `non-test-diff-present`
- `post-merge-validation-ran`
- `audit-report-present`

The validation runner records which evidence labels were satisfied or missing.

## Workflow Semantics

### 1. Test-spec phase

Planner emits one or more `test-spec` nodes before implementation nodes. A test-spec node can create:

- shell validators,
- unit/integration tests,
- OpenSpec scenarios,
- static checks,
- review rubrics.

The node completes only after its own validators pass, for example linting the generated scripts or ensuring the acceptance spec mentions required objective terms.

### 2. Test-review phase

Planner may emit a `test-review` node or the controller may validate the `test-spec` node with a review model. This phase checks whether the proposed tests actually encode the objective.

If review fails, controller sends follow-up to the test-spec node. Implementation nodes remain blocked because they depend on approved test artifacts.

### 3. Artifact lock phase

When the test contract is approved, runtime records `artifactLocks` with:

- path,
- sha256,
- source node id,
- source subagent id/session file,
- approved at timestamp.

The lock can be stored in the DAG node metadata or in a dedicated validation artifact table. Existing DAG JSON stays immutable; runtime can persist resolved locks separately.

### 4. Implementation phase

Implementation nodes depend on their test-spec/review nodes. The implementation prompt tells the subagent:

- tests/validators are already approved,
- do not weaken locked artifacts,
- if the test contract is wrong, report `SUBAGENT_BLOCKED: test spec mismatch ...` rather than editing tests.

### 5. Controller validation phase

When implementation self-reports completion, controller validation performs:

1. sync subagent state,
2. verify locked artifacts unchanged,
3. run declared validators when host policy allows,
4. check required evidence,
5. persist command outputs and evidence summary,
6. mark node complete / needsFollowup / blocked / failed.

Declared validators must execute or fail closed. This is already implemented for skipped validators; this change extends the evidence model around it.

### 6. Audit phase and test-gap loop

Audit nodes can return structured outcomes:

- `passed` — implementation and tests are sufficient.
- `implementation_failed` — approved tests are adequate; implementation must be fixed.
- `test_gap` — tests/validators did not encode the objective well enough.
- `policy_blocked` — human decision required.

For `test_gap`, runtime should support a controlled revision path:

```text
audit detects test gap
  -> mark affected implementation nodes as needsFollowup or planned for rerun
  -> reopen or create a revision of the test-spec node
  -> require renewed artifact locks
  -> rerun implementation validation against the new contract
```

Because current DAGs are acyclic, the first implementation can support this as an explicit controller operation (`reopenNode` / `resetDownstream`) rather than automatic cyclic scheduling.

## Decisions

### D1. Keep runtime as DAG consumer, not test generator

**Choice**
- Runtime accepts validation contracts and enforces them.
- Planner/test-writer models generate concrete test nodes, validators, and rule-pack-specific commands.

**Rationale**
- The runtime already has clear boundaries as parser/validator/scheduler/store.
- Test generation requires project context and model judgment, which belongs in planner and project rule packs.

**Alternative rejected**
- Add LLM test generation directly to runtime. Rejected because it would couple runtime to model prompts, project semantics, and provider policy.

### D2. Use phase-expanded DAGs instead of hidden controller-only phases

**Choice**
- Test-spec, review, implementation, and audit are represented as visible DAG nodes when possible.

**Rationale**
- Users can monitor progress and see which phase is blocked.
- The scheduler can reuse existing dependency mechanics.
- Artifacts and evidence have clear provenance.

**Alternative rejected**
- Hide test writing inside controller validation. Rejected because hidden work is harder to monitor, retry, or audit.

### D3. Lock approved tests and validators by hash

**Choice**
- After approval, runtime records artifact path hashes and verifies them during implementation validation.

**Rationale**
- Prevents implementation subagents from accidentally or intentionally weakening tests.
- Allows legitimate test revisions through an explicit test-gap loop.

**Alternative rejected**
- Rely only on prompts saying not to modify tests. Rejected because prompt-only controls are not durable validation.

### D4. Treat test gaps differently from implementation bugs

**Choice**
- Audit can classify failures as `test_gap`, causing test-spec revision and downstream rerun.

**Rationale**
- If the test is wrong, telling the implementation subagent to satisfy it is wasteful or harmful.
- Explicit test-gap classification makes the quality loop visible and auditable.

**Alternative rejected**
- Always send audit failures back to implementation. Rejected because it conflates bad tests with bad code.

### D5. Fail closed for high-risk implementation nodes

**Choice**
- A high-risk `implementation` node cannot pass with empty validators, no approved test contract, and no required evidence.

**Rationale**
- High-risk work is exactly where self-report-only completion is most dangerous.

**Alternative rejected**
- Let final audit catch all issues. Rejected because final audit happens late and cannot always reconstruct missing per-node evidence.

## Runtime Enforcement Rules

Initial generic enforcement rules:

1. If `validators.length > 0` and validators are skipped, validation fails unless an explicit host allow policy is set.
2. If `risk=high` and `kind=implementation`, require at least one of:
   - validators,
   - approved test-spec contract,
   - explicit validation profile with required evidence.
3. If `artifactLocks` are present, every lock must match before completion.
4. If `requiredEvidence` includes diff evidence, validation must inspect branch/worktree diff against the configured base ref.
5. If an audit reports `test_gap`, runtime must not mark the implementation complete until a new approved contract is available and validation reruns.

## Persistence

Possible schema extensions:

- Add optional `kind` and `validation_json` columns to `goal_dag_nodes`.
- Add `goal_validation_evidence` table:
  - `goal_id`, `node_id`, `subagent_id`, `evidence_id`, `status`, `summary`, `details_json`, `created_at`.
- Add `goal_validation_artifacts` table:
  - `goal_id`, `source_node_id`, `path`, `sha256`, `approved_by_node_id`, `approved_at`.

Memory store mirrors the same fields for tests.

## Planner / Rule-Pack Contract

Planner should support a project-local rule pack, for example:

```json
{
  "validationProfiles": {
    "code-move": {
      "requires": ["validators-ran", "locked-artifacts-unchanged", "implementation-diff-present"]
    }
  },
  "packageMoves": {
    "com.example.common.model.foo": {
      "ownerModule": "foo-module",
      "targetPackage": "com.example.foo.model",
      "forbiddenImports": ["com.example.common.model.foo"]
    }
  }
}
```

Planner can expand this into concrete DAG nodes and validators. Runtime only sees the resulting DAG and validation metadata.

## Risks / Trade-offs

- More DAG nodes increase orchestration overhead and token use.
- Test-writer quality still depends on model and reviewer quality; this workflow makes gaps visible but cannot prove completeness.
- Hash locks can be noisy if generated tests contain timestamps or nondeterministic output; test-writer guidance must avoid nondeterministic artifacts.
- Reopening upstream test-spec nodes from final audit is more complex than a simple acyclic DAG. Initial implementation should require explicit controller operations rather than automatic cyclic execution.
- Project rule packs can become stale; final audit should include a rule-pack freshness check.

## Migration Plan

1. Keep existing DAGs valid. Nodes without `kind` or `validation` use current behavior plus fail-closed skipped validators.
2. Add schema/types/store support for optional validation contracts and evidence.
3. Add validation runner checks for artifact locks, high-risk empty validation, and required evidence.
4. Add monitor/status rendering for validation contracts and latest evidence.
5. Add example DAGs showing test-spec -> implementation -> audit phases.
6. Coordinate a separate `agent-goal-planner` update to generate phase-expanded DAGs from validation profiles and project rule packs.
7. Migrate high-risk project DAGs to the new workflow incrementally.

## Open Questions

- Should runtime expose `/goal node reset <node>` and `/goal node reopen --downstream <node>` as first-class commands for audit-driven test-gap loops?
- Should artifact locks live only in the store, or should the approved lock set be materialized into a checked-in manifest file?
- How strict should default high-risk policy be for legacy DAGs that do not declare `kind`?
- Should final audit be mandatory for all multi-node implementation goals, or only when the planner emits an audit node?
- Should subagent branch integration/merge become part of this change, or remain a follow-up validation evidence feature?
