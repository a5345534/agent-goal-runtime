agent-goal-runtime

Goal / Controller / Node / Subagent 運作邏輯

這份文件說明目前 runtime 如何把一個 Goal 拆成 DAG nodes、由 controller 固定程序推進、監控與預處理 subagent runner/session 狀態，並在 stale runner、缺 session transcript、validation 失敗或 integration 阻塞時做 deterministic recovery。

Goal 是什麼？

Goal 是使用者的高階目標與持久化狀態。它包含 objective、activity state、DAG nodes、subagent attempts、ledger history，以及必要時的 execution workspace / promotion target。

Controller 是什麼？

Controller 是固定程序，不是 LLM agent。它只做排程、同步、validation、integration、recovery、closeout guard，不自行語意改程式。

Node 是什麼？

Node 是 DAG 裡可排程的工作單元，包含 objective、dependencies、expected outputs、validators、completion gates、workspace binding 與模型設定。

Subagent 是什麼？

Subagent 是某個 node 的一次執行 attempt。它有 harness session、session JSONL、workspace、branch、prompt history、retry count 與 integration state。

Controller 每次 poll 的固定流程

1. 讀 durable orchestration state→ 2. sync subagents→ 3. reconcile outcomes→ 4. validation / integration→ 5. start ready nodes→ 6. append controller ledger

所有決策都寫回 store，因此 controller crash/restart 後可以從 DB 與 session files 繼續。

Controller 監控與預處理

檢查項固定處理結果
runner live + session file missing視為啟動中 grace 狀態，繼續等待下一次 poll。starting / running
runner dead + session file missing判定為 non-resumable stale attempt；舊 attempt terminalize，retry budget 內開 replacement subagent。replacement started
session file exists + transcript stale after tool result保留同一 session，送 stale continuation follow-up。same-session recovery
explicit SUBAGENT_RESULT進入 controller validation；validators 必跑。selfReportedComplete → validating
explicit SUBAGENT_BLOCKED若 goal active 且 retry budget 未用完，先做 same-session blocked-node recovery；provider/quota 類直接 blocked。followup or blocked
validation failed有 follow-up prompt 時送回 subagent；相同 failure 超過 cap 後 blocked。needsFollowup / blocked
native-git integration neededvalidation passed 後 merge subagent branch into controller branch；dirty/conflict fail closed。complete 或 blocked

Node 生命週期

planned → ready → running → selfReportedComplete/controllerValidating
          ↘ needsFollowup ↗       ↘ complete
                               ↘ blocked / failed / superseded
  • planned：尚未滿足 dependencies 或尚未排程。
  • ready：dependencies 已完成，queue policy 允許啟動。
  • running：已有 non-terminal subagent attempt。
  • needsFollowup：controller 需要 subagent 補結果 marker 或修 validation failure。
  • controllerValidating：subagent 已回報完成，controller 正在跑 deterministic validators / gates。
  • complete：validators / integration gates 完成。
  • blocked / failed：retry 或外部條件阻止繼續；active goal 可對部分 blocked 狀態做 bounded recovery。

Subagent 生命週期

planned → workspaceCreated → sessionStarted → running ↔ idle
                                      ↓              ↓
                               needsFollowup ← selfReportedComplete
                                      ↓              ↓
                                  blocked/failed → controllerValidating → complete
  • sessionStarted / running 不再單獨代表 runner live；Pi adapter 會看可驗證仍 alive 的 in-memory handle 或 background-runner PID inventory。
  • missing session replacement：如果 session JSONL 不存在且 runner 已死，controller 不會對不存在的 session 送 prompt，而是開新的 replacement attempt。
  • same-session recovery：只在 transcript 存在且可 resume 時使用，避免丟失上下文。
  • retryCount 會限制自動修復，避免無限重啟或 retry storm。

完成與 closeout 語意

Goal 的 complete 不只是 controller branch 上的 nodes 完成。對 auto Pi/native-git goal，complete 必須代表：

  • 所有 required DAG nodes 都達到 complete；
  • repository-changing subagent branches 已 integration；
  • controller branch promotion/merge 到 target ref 成功；
  • cleanup 在 promotion 成功後才執行；
  • dirty worktree、merge conflict、blocked/failed required nodes 都會 fail closed。

因此 subagent 自己說「完成」只是 evidence；真正完成由 controller validation + integration + promotion guard 決定。

目前新增的 stale runner/session 控制

if subagent is syncable:
  state = adapter.getSessionState(subagent)

Pi adapter liveness:
  live = verifiedInMemoryHandlePidAlive(subagent) || liveBackgroundRunnerPidExists(subagent)

if sessionFile missing and live:
  report starting

if sessionFile missing and !live:
  report failed: session file not found

controller failed recovery:
  if error matches missing-session:
    if retryCount < maxAutoRetries:
      mark old attempt failed
      start replacement subagent in same workspace/branch when available
    else:
      block node and subagent with diagnostic
